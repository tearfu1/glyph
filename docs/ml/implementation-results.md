# Реализация ML-компонента платформы Glyph

## 3.1. Архитектура ML-сервиса

ML-сервис реализован как отдельный микросервис на FastAPI (Python), взаимодействующий с основным бэкендом (Rust/Axum) по HTTP. Сервис развёрнут в Docker Compose совместно с векторной базой данных Qdrant.

Архитектура пайплайна генерации ответа:

```
Вопрос пользователя
    → Embedding (multilingual-e5-base) → вектор запроса
    → Qdrant (cosine similarity, фильтр по автору) → top-5 релевантных фрагментов
    → rugpt3small + LoRA-адаптер автора → генерация ответа в стиле автора
    → JSON-ответ с текстом и источниками
```

## 3.2. Сбор и подготовка корпуса

Корпус собран из открытых источников (lib.ru, произведения в public domain). Реализован скрипт `download_texts.py`, осуществляющий:
- HTTP-запросы с User-Agent и задержкой 2 секунды между запросами
- Парсинг HTML-страниц lib.ru (кодировка windows-1251, извлечение текста из вложенных `<dd>` тегов через BeautifulSoup)
- Идемпотентное сохранение (повторный запуск не перезагружает существующие файлы)

## 3.3. Препроцессинг и чанкинг

Скрипт `preprocess.py` выполняет двухэтапную обработку:

**Очистка текста:**
- Удаление HTML-сущностей (`&nbsp;`, `&#...;`)
- Удаление служебных пометок и сносок (`[1]`, `[2]`)
- Нормализация пробелов и переносов строк

**Чанкинг** — разбиение на фрагменты фиксированного размера с использованием GPT-2 BPE-токенизатора (совместимого с rugpt3small):
- Размер чанка: 256 токенов (~150-200 слов)
- Перекрытие (overlap): 64 токена (25%)
- Формат хранения: JSONL с метаданными (автор, произведение, номер чанка)

Выбор 256 токенов обусловлен спецификой художественных текстов: абзацы в литературе короче, чем в технической документации, и 256 токенов (~1-2 абзаца) образуют семантически целостную единицу. Перекрытие компенсирует разрывы контекста на границах.

|-------|--------|
| Достоевский | 43 276 |
| Чехов | 50 071 |
| Булгаков | 5 283 |
| **Итого** | **98 630** |

## 3.4. Векторная индексация (RAG)

Для реализации Retrieval-Augmented Generation каждый чанк преобразован в 768-мерный вектор с помощью модели `intfloat/multilingual-e5-base` — мультиязычного Transformer-энкодера, хорошо работающего с русскоязычными текстами.

Особенность модели e5: требует префиксы `"query: "` для запросов и `"passage: "` для документов, что улучшает качество retrieval за счёт асимметричного обучения.

Векторы загружены в Qdrant — векторную базу данных, обеспечивающую:
- Approximate Nearest Neighbor поиск через HNSW-индекс
- Фильтрацию по метаданным (автор, произведение)
- Косинусную меру близости

Коллекция `glyph_chunks`: 98 630 точек, метрика Cosine.

Верификация: запрос «Почему Раскольников убил старуху?» с фильтром по Достоевскому возвращает top-5 фрагментов из «Преступления и наказания», непосредственно описывающих убийство, с cosine similarity 0.85-0.86.

## 3.5. LoRA fine-tuning

**Базовая модель:** `ai-forever/rugpt3small_based_on_gpt2` (125M параметров), предобученная на большом русскоязычном корпусе.

**Метод:** LoRA (Low-Rank Adaptation) — вместо обновления всех 125M параметров, замораживаются исходные веса и добавляются малоранговые обучаемые матрицы:

$$W = W_0 + \Delta W = W_0 + BA$$

где $B \in \mathbb{R}^{d \times r}$, $A \in \mathbb{R}^{r \times k}$, $r \ll \min(d, k)$.

При $r = 8$, $d = k = 768$: обучается $8 \times (768 + 768) = 12\,288$ параметров на слой вместо $768 \times 768 = 589\,824$ — **сокращение в 48 раз**.

**Целевые модули:** `c_attn` и `c_proj` (attention-слои GPT-2).

**Подготовка данных:** все тексты автора конкатенируются, токенизируются и нарезаются на блоки по 512 токенов. Labels = input_ids (авторегрессионное обучение). Разбиение 90/10 на train/eval.

**Сетка гиперпараметров:**

| Эксперимент | r | alpha | lr | epochs | Обучаемые параметры | % от общего |
|-------------|---|-------|----|--------|--------------------:|------------:|
| E1_default | 8 | 16 | 2e-4 | 5 | 811 008 | 0.49% |
| E2_low_rank | 4 | 8 | 2e-4 | 5 | 405 504 | 0.25% |
| E3_high_rank | 16 | 32 | 2e-4 | 5 | 1 622 016 | 0.98% |
| E4_long_train | 8 | 16 | 1e-4 | 10 | 811 008 | 0.49% |

Общее количество параметров базовой модели: 164 645 376. Batch size = 8, fp16, warmup = 100 steps.

**Анализ результатов:**

1. **E3_high_rank (r=16) — лучший для всех авторов** по eval loss. Увеличение ранга с 8 до 16 даёт стабильное улучшение на 0.5-0.6%, что свидетельствует о том, что стилистические паттерны авторов требуют больше обучаемых параметров для адекватного захвата.

2. **E2_low_rank (r=4) — худший результат.** Слишком мало параметров (0.25%) недостаточно для передачи авторского стиля.

3. **E4_long_train (10 эпох) не улучшил результаты** по сравнению с E1 (5 эпох), что говорит о том, что модель сходится уже к 5-й эпохе, а дополнительные эпохи не дают значимого выигрыша.

4. **Eval loss ниже train loss** у всех экспериментов — переобучение отсутствует.

5. **Чехов показывает наименьший loss** (3.140), что объясняется наибольшим объёмом корпуса (9.2 млн символов) и более однородным стилем коротких рассказов.

Размер каждого адаптера: E1/E4 — ~3.2 MB, E2 — ~1.6 MB, E3 — ~6.4 MB. Для инференса выбран **E3_high_rank** как лучший по eval loss.

## 3.6. Inference pipeline

Реализованы модули:

- **Embedder** — обёртка над SentenceTransformer для создания эмбеддингов запросов
- **Retriever** — поиск в Qdrant с фильтрацией по автору
- **ModelManager** — загрузка базовой модели и LoRA-адаптеров с кешированием
- **AnswerGenerator** — формирование промпта с контекстом и генерация ответа

Промпт-шаблон:
```
Контекст из произведений автора:
{top-5 релевантных фрагментов}

Вопрос читателя: {вопрос}

Ответ в стиле автора:
```

Параметры генерации: temperature=0.8, top_p=0.9, repetition_penalty=1.2, max_new_tokens=256.

Docker Compose с 4 сервисами:

- Rust/Axum backend — REST API
- Qdrant v1.9 — векторная БД
- Python/FastAPI ml-service — ML-пайплайн

## 3.9. Оценка качества (evaluation pipeline)

Реализован модуль `src/evaluation/`, содержащий три независимые метрики. Все метрики рассчитываются для 6 условий: baseline (rugpt3small без RAG), rag_only (RAG без LoRA), rag_lora с LoRA E2/E1/E3 (ранг 4/8/16) и rag_lora_instruct с E5_instruct.

### 3.9.1. Perplexity

PPL рассчитывается на held-out 10% корпуса автора (воспроизводимый split seed=42, block_size=512). Формула: $\text{PPL} = \exp(\overline{\mathcal{L}_{CE}})$ на валидационных блоках.

**Результаты:**

| Автор | Baseline | E1 (r=8) | E2 (r=4) | **E3 (r=16)** | E4 (long) | E5_instruct |
|-------|---------:|---------:|---------:|--------------:|----------:|------------:|
| Достоевский | 32.96 | 29.20 | 29.65 | **28.68 (−13.0%)** | 29.21 | 45.39 (+37.7%) |
| Чехов | 27.18 | 23.71 | 24.12 | **23.26 (−14.4%)** | 23.72 | 37.59 (+38.3%) |
| Булгаков | 29.99 | 27.56 | 27.85 | **27.18 (−9.4%)** | 27.53 | 41.63 (+38.8%) |

**Анализ:** LoRA, обученная на сыром тексте (E1-E4), улучшает PPL на 9-14% относительно baseline, причём E3 (r=16) — победитель у всех трёх авторов. E5_instruct, обученный на QA-формате, «проигрывает» по PPL на 37-39% — это классический *alignment tax*, поскольку модель теряет способность предсказывать сырой текст в обмен на формат «вопрос-ответ».

### 3.9.2. Semantic similarity

Cosine между embedding-ом сгенерированного ответа и embedding-ами извлечённых top-5 контекстных чанков (через multilingual-e5-base).

| Автор | rag_only | r=4 | r=8 | r=16 | E5_instruct |
|-------|---------:|----:|----:|-----:|------------:|
| Достоевский | 0.820 | 0.822 | 0.815 | 0.822 | 0.822 |
| Чехов | 0.820 | 0.812 | 0.811 | 0.819 | 0.812 |
| Булгаков | 0.814 | 0.812 | 0.809 | 0.812 | 0.802 |

Все условия близки (0.80-0.83): retrieval уже задаёт семантическое поле, LoRA-адаптация мало на него влияет.

### 3.9.3. Стилевые метрики

Для каждого условия вычислены TTR (Type-Token Ratio), средняя длина предложения, hapax ratio (доля уникальных слов) и POS-распределение (pymorphy3 + razdel).

Ключевое наблюдение: POS-распределение генераций смещено в сторону большего числа существительных и меньшего числа глаголов по сравнению с оригиналом — это отражает жанровый разрыв (аналитический ответ vs художественная проза), не имеющий отношения к качеству LoRA.

## 3.10. Instruction tuning на synthetic-датасете

Этап B-D дополнил основную работу экспериментом с *instruction tuning* — обучением LoRA не на сыром корпусе, а на синтетических парах «вопрос→ответ в стиле автора».

### 3.10.1. Генерация датасета

Скрипт `generate_synthetic_dataset.py` реализует универсальный клиент к LLM (поддерживаются Groq, Gemini, Anthropic и корпоративный LiteLLM-прокси с OpenAI-compatible API). Для каждого случайно выбранного фрагмента корпуса LLM генерирует пару (вопрос, ответ). Используемая модель — GPT-5.4-nano через LiteLLM.

Параметры промпта: вопрос 1-2 предложения (15-30 слов), ответ 2-3 полноценных предложения (40-80 слов). Датасет идемпотентен — повторный запуск продолжает с места остановки.

**Итоговый датасет:** 1497 пар (500 + 497 + 500), средняя длина вопроса ~26 слов, ответа ~68 слов, охват — все 42 произведения трёх авторов. Размер ~4.5 MB JSONL.

### 3.10.2. Instruction fine-tuning

Скрипт `train_instruct_lora.py` обучает LoRA с гиперпараметрами победившего E3 (r=16, α=32, 5 эпох, lr=2e-4), но:
- **Входной формат:** `Вопрос: {q}\nОтвет: {a}`
- **Маскирование префикса:** токены вопроса получают label=-100, loss считается только на ответе
- **Блок:** max_length=384 токена, dynamic padding

Обучение выполнено на Kaggle T4 x2 за ~50 минут. Результирующие адаптеры `E5_instruct` имеют размер 6.3 MB каждый.

**Eval loss (на Q&A split):**

| Автор | train_loss | eval_loss |
|-------|----------:|----------:|
| Достоевский | 3.93 | 3.59 |
| Чехов | 3.94 | 3.65 |
| Булгаков | 4.04 | 3.78 |

Eval loss < train loss у всех авторов (HuggingFace возвращает training_loss как среднее по всем эпохам), что говорит об отсутствии переобучения.

### 3.10.3. Качественное сравнение E5_instruct vs E1-E4

| Свойство | LoRA на прозе (E1-E4) | LoRA instruct (E5) |
|----------|----------------------|--------------------|
| Средняя длина ответа | 120-165 слов | 41-83 слова |
| Попадание в тему вопроса | Редко | Чаще |
| Галлюцинации имён/дат | Часто | Значительно реже |
| Воспроизведение стиля автора | Есть попытки | Сглаживается |

E5_instruct демонстрирует **alignment tax**: модель теряет часть языкового моделирования (высокая PPL), но выигрывает в соблюдении формата и точности ответа. Для продакшена это предпочтительный режим — именно он выбран в `ModelManager` как default.

## 3.12. Два режима работы системы

В результате проведённых экспериментов в системе реализованы **два независимых режима генерации ответов**, каждый из которых адресует разную задачу и выбирается заменой LoRA-адаптера без каких-либо иных изменений пайплайна.

### 3.12.1. Стилистический режим (E1-E4)

**Назначение:** имитация литературного стиля автора — модель «продолжает писать» как автор, пытаясь воспроизвести его лексику, синтаксис, интонацию.

**Как устроено:**
- Базовая модель rugpt3small + LoRA, обученная на сыром корпусе автора (continued pretraining)
- Вход: промпт с контекстом и вопросом, выход: длинный (120-165 слов) нарратив
- Loss на обучении — стандартный causal LM loss на художественном тексте

**Сильные стороны:** богатая лексика, живая проза, попытка воспроизвести голос автора.
**Слабые:** галлюцинирует фактами, имена-выдумки, часто уходит от темы.

### 3.12.2. Функциональный режим (E5_instruct)

**Назначение:** прямой ответ на вопрос читателя — по теме, с опорой на контекст, без художественного украшательства.

**Как устроено:**
- Та же базовая модель + LoRA, но обученная на **синтетическом датасете «вопрос-ответ»** (1497 пар)
- Формат обучения: `Вопрос: {q}\nОтвет: {a}` с маскированием префикса — loss только на токенах ответа
- Вход: RAG-контекст + вопрос, выход: короткий (40-80 слов) ответ по теме

**Сильные стороны:** попадание в тему, отсутствие галлюцинаций имён и дат, компактность.
**Слабые:** теряется авторский стиль — модель отвечает «нейтральным аналитическим тоном».

### 3.12.3. Сравнение режимов

| Критерий | Стилистический (E3) | Функциональный (E5_instruct) |
|----------|---------------------|------------------------------|
| Perplexity (Достоевский) | 28.7 (−13% vs baseline) | 45.4 (+38%) |
| Средняя длина ответа | 120-165 слов | 41-83 слова |
| Попадание в тему | Редко | Часто |
| Галлюцинации фактов | Часто | Значительно реже |
| Воспроизведение стиля | Есть | Минимально |

Расхождение в PPL — не ошибка экспериментов, а демонстрация **alignment tax**: instruction tuning сдвигает оптимизационную цель модели с «продолжения прозы» на «соблюдения формата QA», и это неизбежно ухудшает метрику первого типа. Ни один режим не является «правильным» в абсолютном смысле — они адресуют разные пользовательские сценарии.

**В продакшене выбран функциональный режим (E5_instruct)** — для массового пользователя пригодность ответа и отсутствие галлюцинаций важнее стилистической имитации. Переключение на стилистический режим тривиально — изменение приоритета в `ModelManager.load_adapter`.

## 3.13. Production deployment

Финальная интеграция работает end-to-end:

```
Vue frontend (5173) → Rust Axum backend (3000) → FastAPI ML-service (8001)
                                                    ↓
                                        Qdrant (6333) + E5_instruct LoRA
```

`ModelManager` в production-инференсе загружает адаптеры с приоритетом `E5_instruct → E3_high_rank → E1_default`. В `api/routes.py` реализован `resolve_author_slug()` с картой русских имён (Достоевский/Чехов/Булгаков → dostoevsky/chekhov/bulgakov), поскольку бэкенд передаёт имена в полном виде из `up_user.full_name`.

Пример ответа E5_instruct на вопрос «Почему Раскольников решился убить старуху-процентщицу?»:

> «Романы пишут о том же самом человеке как об убийце; но они говорят лишь про то, чтобы оправдать смерть». И потому здесь звучит уже совсем другая мысль: убийца должен быть убит так жестоко, будто бы это было сделано нарочно для оправдания смерти человека. Но ведь именно эта идея кажется ему самой страшной правдой жизни, когда она оказывается ложной по отношению ко всему человеческому существу.

## 3.14. Ограничения и направления развития

1. **Масштаб базовой модели.** rugpt3small содержит 125M параметров — это в 1400 раз меньше GPT-4. Никакая fine-tuning не превратит её в безошибочную QA-систему. Для продакшен-качества необходима замена на 7B+ модель (Saiga Mistral, Llama 3 + русский fine-tune) с той же методологией RAG+LoRA+instruction tuning.
2. **Стилевой vs функциональный компромисс.** E3 лучше воспроизводит авторский стиль, но галлюцинирует; E5 корректнее отвечает, но теряет голос автора. Возможное улучшение — ансамбль / переключение по типу запроса.
3. **Оценка с живыми ассессорами.** Автоматические метрики (PPL, semantic, style) не заменяют субъективную оценку читателем. Для полной картины требуется blind-сравнение сгенерированных ответов носителями языка.
4. **Расширение корпуса.** Система спроектирована универсально — добавление нового автора сводится к запуску пайплайна download → preprocess → chunk → embed → train для новых текстов.
