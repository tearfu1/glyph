# Реализация ML-компонента платформы Glyph

## 3.1. Архитектура ML-сервиса

ML-сервис реализован как отдельный микросервис на FastAPI (Python), взаимодействующий с основным бэкендом (Rust/Axum) по HTTP. Сервис развёрнут в Docker Compose совместно с векторной базой данных Qdrant.

Архитектура пайплайна генерации ответа:

```
Вопрос пользователя
    → Embedding (multilingual-e5-base) → вектор запроса
    → Qdrant (cosine similarity, фильтр по автору) → top-5 релевантных фрагментов
    → rugpt3small + LoRA-адаптер автора → генерация ответа в стиле автора
    → JSON-ответ с текстом и источниками
```

## 3.2. Сбор и подготовка корпуса

Корпус собран из открытых источников (lib.ru, произведения в public domain). Реализован скрипт `download_texts.py`, осуществляющий:
- HTTP-запросы с User-Agent и задержкой 2 секунды между запросами
- Парсинг HTML-страниц lib.ru (кодировка windows-1251, извлечение текста из вложенных `<dd>` тегов через BeautifulSoup)
- Идемпотентное сохранение (повторный запуск не перезагружает существующие файлы)

## 3.3. Препроцессинг и чанкинг

Скрипт `preprocess.py` выполняет двухэтапную обработку:

**Очистка текста:**
- Удаление HTML-сущностей (`&nbsp;`, `&#...;`)
- Удаление служебных пометок и сносок (`[1]`, `[2]`)
- Нормализация пробелов и переносов строк

**Чанкинг** — разбиение на фрагменты фиксированного размера с использованием GPT-2 BPE-токенизатора (совместимого с rugpt3small):
- Размер чанка: 256 токенов (~150-200 слов)
- Перекрытие (overlap): 64 токена (25%)
- Формат хранения: JSONL с метаданными (автор, произведение, номер чанка)

Выбор 256 токенов обусловлен спецификой художественных текстов: абзацы в литературе короче, чем в технической документации, и 256 токенов (~1-2 абзаца) образуют семантически целостную единицу. Перекрытие компенсирует разрывы контекста на границах.

Результат:

| Автор | Чанков |
|-------|--------|
| Достоевский | 43 276 |
| Чехов | 50 071 |
| Булгаков | 5 283 |
| **Итого** | **98 630** |

## 3.4. Векторная индексация (RAG)

Для реализации Retrieval-Augmented Generation каждый чанк преобразован в 768-мерный вектор с помощью модели `intfloat/multilingual-e5-base` — мультиязычного Transformer-энкодера, хорошо работающего с русскоязычными текстами.

Особенность модели e5: требует префиксы `"query: "` для запросов и `"passage: "` для документов, что улучшает качество retrieval за счёт асимметричного обучения.

Векторы загружены в Qdrant — векторную базу данных, обеспечивающую:
- Approximate Nearest Neighbor поиск через HNSW-индекс
- Фильтрацию по метаданным (автор, произведение)
- Косинусную меру близости

Коллекция `glyph_chunks`: 98 630 точек, метрика Cosine.

Верификация: запрос «Почему Раскольников убил старуху?» с фильтром по Достоевскому возвращает top-5 фрагментов из «Преступления и наказания», непосредственно описывающих убийство, с cosine similarity 0.85-0.86.

## 3.5. LoRA fine-tuning

**Базовая модель:** `ai-forever/rugpt3small_based_on_gpt2` (125M параметров), предобученная на большом русскоязычном корпусе.

**Метод:** LoRA (Low-Rank Adaptation) — вместо обновления всех 125M параметров, замораживаются исходные веса и добавляются малоранговые обучаемые матрицы:

$$W = W_0 + \Delta W = W_0 + BA$$

где $B \in \mathbb{R}^{d \times r}$, $A \in \mathbb{R}^{r \times k}$, $r \ll \min(d, k)$.

При $r = 8$, $d = k = 768$: обучается $8 \times (768 + 768) = 12\,288$ параметров на слой вместо $768 \times 768 = 589\,824$ — **сокращение в 48 раз**.

**Целевые модули:** `c_attn` и `c_proj` (attention-слои GPT-2).

**Подготовка данных:** все тексты автора конкатенируются, токенизируются и нарезаются на блоки по 512 токенов. Labels = input_ids (авторегрессионное обучение). Разбиение 90/10 на train/eval.

**Сетка гиперпараметров:**

| Эксперимент | r | alpha | lr | epochs | Обучаемые параметры | % от общего |
|-------------|---|-------|----|--------|--------------------:|------------:|
| E1_default | 8 | 16 | 2e-4 | 5 | 811 008 | 0.49% |
| E2_low_rank | 4 | 8 | 2e-4 | 5 | 405 504 | 0.25% |
| E3_high_rank | 16 | 32 | 2e-4 | 5 | 1 622 016 | 0.98% |
| E4_long_train | 8 | 16 | 1e-4 | 10 | 811 008 | 0.49% |

Общее количество параметров базовой модели: 164 645 376. Batch size = 8, fp16, warmup = 100 steps.

Train/eval samples: Достоевский — 3501/390, Чехов — 4298/478, Булгаков — 459/52.

**Анализ результатов:**

1. **E3_high_rank (r=16) — лучший для всех авторов** по eval loss. Увеличение ранга с 8 до 16 даёт стабильное улучшение на 0.5-0.6%, что свидетельствует о том, что стилистические паттерны авторов требуют больше обучаемых параметров для адекватного захвата.

2. **E2_low_rank (r=4) — худший результат.** Слишком мало параметров (0.25%) недостаточно для передачи авторского стиля.

3. **E4_long_train (10 эпох) не улучшил результаты** по сравнению с E1 (5 эпох), что говорит о том, что модель сходится уже к 5-й эпохе, а дополнительные эпохи не дают значимого выигрыша.

4. **Eval loss ниже train loss** у всех экспериментов — переобучение отсутствует.

5. **Чехов показывает наименьший loss** (3.140), что объясняется наибольшим объёмом корпуса (9.2 млн символов) и более однородным стилем коротких рассказов.

Размер каждого адаптера: E1/E4 — ~3.2 MB, E2 — ~1.6 MB, E3 — ~6.4 MB. Для инференса выбран **E3_high_rank** как лучший по eval loss.

## 3.6. Inference pipeline

Реализованы модули:

- **Embedder** — обёртка над SentenceTransformer для создания эмбеддингов запросов
- **Retriever** — поиск в Qdrant с фильтрацией по автору
- **ModelManager** — загрузка базовой модели и LoRA-адаптеров с кешированием
- **AnswerGenerator** — формирование промпта с контекстом и генерация ответа

Промпт-шаблон:
```
Контекст из произведений автора:
{top-5 релевантных фрагментов}

Вопрос читателя: {вопрос}

Ответ в стиле автора:
```

Параметры генерации: temperature=0.8, top_p=0.9, repetition_penalty=1.2, max_new_tokens=256.

## 3.7. API ML-сервиса

```
GET  /health                → {"status": "ok", "models_loaded": true}
POST /api/generate-answer   → {"answer": "...", "sources": [...]}
```

Request: `{"question": str, "author": str}`
Response: `{"answer": str, "sources": [{"text": str, "score": float, "book": str}]}`

## 3.8. Инфраструктура

Docker Compose с 4 сервисами:
- PostgreSQL 16 — основная БД
- Rust/Axum backend — REST API
- Qdrant v1.9 — векторная БД
- Python/FastAPI ml-service — ML-пайплайн

ML-сервис использует CPU-only PyTorch для инференса. Время генерации ответа: ~20-30 секунд на CPU. При старте сервис загружает embedder (~1 GB), базовую модель (~0.5 GB) и все 3 LoRA-адаптера (~10 MB). HuggingFace-модели кешируются в Docker volume для ускорения перезапусков.
